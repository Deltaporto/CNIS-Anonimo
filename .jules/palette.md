## 2024-05-18 - Making Custom DIV Interactive Elements Accessible
**Learning:** When using generic elements like `div` for core interactions (e.g., drag-and-drop upload zones), adding a click listener is not enough for keyboard users. They need `role="button"`, `tabindex="0"`, a keyboard event listener for `Enter` and `Space`, and an explicit focus style (`:focus-visible`) to be fully accessible. Adding an explicit `aria-label` also significantly improves the screen reader experience.
**Action:** Always verify if a custom interactive element can be reached via the `Tab` key and triggered via the keyboard, and check if decorative SVGs within it are hidden from screen readers using `aria-hidden="true"`.
## 2024-05-19 - Making Dynamic Process States Accessible and Safe
**Learning:** For long-running synchronous client-side actions (like JSZip generation), failing to lock the UI can cause users to click multiple times, freezing the thread further. Simply changing the button text or adding a spinner isn't enough; the button MUST be explicitly disabled, wrapped in a `try...finally` block to guarantee restoration of state, and dynamic text elements representing processing state (like the file item's status span) need `role="status"` and `aria-live="polite"` so screen readers are aware of the processing phases.
**Action:** Always wrap heavy client side actions with a Try/Finally that manages a disabled UI state, and ensure dynamic status changes are communicated to screen readers via aria-live regions.
